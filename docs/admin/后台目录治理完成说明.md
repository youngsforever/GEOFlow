# 后台目录治理完成说明

本文用于说明当前后台目录的分层结果、访问约定和后续维护规则。

## 当前结论

后台已经完成从 `/admin/...` 到 `/geo_admin/...` 的主链路切换，并将后台目录整理为三层：

1. 正式功能页
2. 兼容入口壳
3. 历史源码目录 `admin/legacy/`

这样做的目标是：

- 正式功能入口唯一化
- 历史页面不再混入主链路
- 旧链接仍然可兼容访问
- 后续维护时更容易判断哪些文件可以继续演进，哪些只能保留兼容

## 一、正式功能页

正式功能页仍位于 `admin/` 根目录，但对外访问统一走：

- `/geo_admin/`
- `/geo_admin/dashboard.php`
- `/geo_admin/tasks.php`
- `/geo_admin/articles.php`
- `/geo_admin/materials.php`
- `/geo_admin/ai-configurator.php`
- `/geo_admin/site-settings.php`
- `/geo_admin/security-settings.php`

这些页面是真正承载业务逻辑的后台入口，后续功能开发应优先接入这里。

## 二、兼容入口壳

部分旧页面仍保留在 `admin/` 根目录，但它们已经不再承载完整业务逻辑，只负责跳转到正式页。

### 过渡入口壳

- `materials-new.php` -> `materials.php`
- `tasks-new.php` -> `tasks.php`
- `articles-new.php` -> `articles.php`
- `authors-new.php` -> `authors.php`
- `ai-config-new.php` -> `ai-configurator.php`
- `login-new.php` -> `/geo_admin/`

### 历史兼容壳

- `ai-config-backup.php` -> `ai-configurator.php`
- `ai-config-simple.php` -> `ai-configurator.php`
- `dashboard-backup.php` -> `dashboard.php`
- `dashboard-simple.php` -> `dashboard.php`
- `tasks-safe.php` -> `tasks.php`

### 测试页兼容壳

- `minimal-test.php`
- `simple-test.php`
- `test-admin.php`
- `test-dashboard.php`
- `test-fixes.php`
- `test-navigation.php`
- `verify-fixes.php`

这些文件原路径还保留，是为了避免旧书签、旧文档或历史引用直接 404，但它们本身已经不应继续扩展业务逻辑。

## 三、历史源码目录

历史页面和测试页面的旧实现，已集中收纳到：

- `admin/legacy/`

其中包括：

- 历史兼容页旧实现
  - `ai-config-backup.php`
  - `ai-config-simple.php`
  - `dashboard-backup.php`
  - `dashboard-simple.php`
  - `tasks-safe.php`
- 测试 / 验证页旧实现
  - `minimal-test.php`
  - `simple-test.php`
  - `test-admin.php`
  - `test-dashboard.php`
  - `test-fixes.php`
  - `test-navigation.php`
  - `verify-fixes.php`

这些文件只用于历史排查、问题回溯或比对旧实现，不应再作为正式维护入口。

## 四、路径规则

后台对外 URL 约定如下：

- 正式后台统一使用 `/geo_admin/...`
- 旧 `/admin/...` 仍保留兼容跳转
- 后台内部生成链接统一通过：
  - `admin_url()`
  - `admin_redirect()`
- 后台前端 JS 请求统一通过：
  - `window.ADMIN_BASE_PATH`
  - `window.adminUrl()`

禁止继续新增：

- 写死的 `/admin/...`
- 写死的 `/geo_admin/...`
- 相对路径 `fetch('xxx.php')` 作为长期方案

## 五、维护约定

后续维护时遵守以下规则：

1. 新功能只接入正式功能页  
不要再往 `simple / backup / new / safe / test / verify` 文件里加业务逻辑。

2. 兼容壳只做跳转  
兼容壳文件应保持极薄，不承担数据库查询、页面渲染或复杂逻辑。

3. 历史源码不作为主修改目标  
如需参考旧实现，应查看 `admin/legacy/`，但不要默认在那里继续迭代功能。

4. 后台路径改动必须走统一 helper  
PHP 用 `admin_url()` / `admin_redirect()`，前端 JS 用 `window.adminUrl()`。

## 六、当前收益

这轮治理完成后，后台的直接收益是：

- `/geo_admin/...` 成为清晰的正式后台入口
- 后台样式与模板引用不再因为路径变化而串到前台
- 正式页、兼容页、历史页边界明确
- 后续继续做权限、导航、AJAX、部署调整时，修改范围更可控

## 七、后续建议

如果继续收尾，建议按这个顺序：

1. 定期检查 `admin/legacy/` 是否还有保留价值
2. 在文档和部署说明中只保留 `/geo_admin/...`
3. 后续新增后台模块时，直接登记到正式页面清单中
